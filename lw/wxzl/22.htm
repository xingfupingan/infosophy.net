那年夏天，离现在大约是两年前，一个好朋友告诉我，由于他在Gosling Emacs（小鹅公司的宏编辑器）早期开发中的工作，他有了来自GOSLING公司对他的许可――在一个通知中，派他去分发那个程序的他的版本。最初，GOSLING公司一直支持他的宏编辑器，一直免费分发它并吸引了很多人来帮助开发它，按照他自己手册中表达的，基于GOSLING公司自己承诺的意愿，他将遵循我起动那个原始宏编辑器的同样精神。然而，后来他又给每个人的背上狠狠来了一刀――通过在它上面加上版权，让人承诺不再分发，然后把它卖给了一个软件服务公司（译注：软件服务公司，software house，直译软件屋；向用户提供软件支持服务的一种公司。服务范围从提供使用手册、其它相关资料、建议与策划到部分计算机程序设计服务）。正如你从这个经历中会推测到的那样，在我后来与他的各种私人交道中，他在每一方面都表现出怯懦与卑劣。

但不管怎样，我的朋友给了我这个程序，而我的目的是在最高水平上改变那些编辑命令，以使它们与我过去熟悉的原始宏编辑器兼容。对于让它们操作数值变元（numerical arguments）的全部结合等等，人们可能指望它们能操作并拥有我想要的所有属性。但这之后我很快发现，那编辑器的扩展语言，叫做MOCKLISP（仿LISP，伪LISP），对执行此任务并不足够。我觉得我得立即替换它，以便做我计划着要做的事。虽然我早就有了或许哪天要用真LISP替换MOCKLISP的念头，但这会儿我认识到，这必需是首先要完成的。好的，MOCKLISP被叫作MOCK（仿）的理由，是它没有结构数据类型：它没有LISP的一些表，它没有任何一种数组。它也没有LISP符号，这是些带名字的对象：对任何特定名称，只有一个对像，因此你可键入名称并总能返回同一个对象。而这些不足极大地妨碍了多种程序的编制，你不得不用复杂的、并非真正是恰当方式的串操作（string-manipulation，字符串处理）来做事。

因此我编写了一个LISP解释程序并把它插入到MOCKLISP中，在运行中我发现，我不得不重写很多编辑器的内部数据结构，因为我要它们成为LISP的对象。我想让LISP与编辑器间的接口界面干净利落，这意味着像编辑器的缓存区、子过程、窗体和缓存区定位等这样一些对象，都必须是LISP的对像，以便作用于（work on）它们的编辑器的基本要素（primitives，或“原函数”）都可以有效调用，就像带LISP数据的LISP函数一样。这就是说，我得重新设计所有这些对象的数据格式，并重新编写所有作用于它们的函数，结果是大约六个月之后，我重写了几乎有关编辑器内部的每样东西。

另外，由于在MOCKLISP（译注：MOCKLISP，一种编程语言）中编写东西是如此艰难，因此所有用MOCKLISP编写的东西都非常含混不清，而通过重新编写它们去获得真LISP的强大优势的话，我就能够使它们变得更加强力高效，以及更简单与更快速。因此我做了这事，而结果是，到我开始分发这个程序时，只有一个小片断保留了我以前接受它时的样子。

At this point, the company that Gosling thinks he sold the program to challenged my friend's right to distribute it, and the message was on backup tapes, so he couldn't find it. And Gosling denied having given him permission.

GOSLING公司认为销售了该程序的那个公侵犯了我朋友的分发权，而有关分发权的信息是在备份（backup，后备）磁盘上，因此他未能发现它。而GOSLING则否认给过他许可权。然后，发生了一件奇怪的事。他正与这个公司谈判，情况看来好像是，该公司主要牵涉到不拥有任何类似于他们正分发着的东西的分发品。你瞧，他仍在分发着，而那个他工作过的公司，它叫“兆测”（MEGATEST），仍在分发他曾已经给了我的同样东西――那确实是一个带有他的修改的“GOSLING宏编辑器”的旧版本，因而，他要与他们签订一个他将分发该版本并转而采用GNU宏编辑器的协议，而届时他们则承认他毕竟还是真正得到了该许可，据信到那时各方都将称心如意。而这个公司正告诉我，说想要分发GNU宏编辑器，当然是自由的（分发），但也销售各种支持服务，他们还想雇请我来帮助干这些工作。有点奇怪的是他们后来又变卦了，拒绝签署那个协议，并在网络上发出了一条信息，说我未获准分发该程序。他们并不具体说他们将做什么事，他们只说尚不清楚他们是否或许某天要做某事。而这就足以让人害怕，致使无人再敢使用它，这是件悲哀的事。

――有时，我认为我一生可以做的最好的事或许是：发现巨量的、商业保密的私权软件，然后在一个街角边开始伸手散发其复制品，让它不再是商业秘密，对我来说，这或许将是比亲自具体编写自由软件更为有效的办法，去提供给人们以新的自由软件，可每个人都胆小得甚至不敢接受它。――

因此我被迫重写所有保留原样的剩余部分，花了大约一周半的时间，我做完了。看来，他们大获全胜。自那以后，我肯定永远不会以任何方式与他们合作了。

GNU宏编辑器之后的时间相当平静，总的说来，那大概有一年半，我开始重新着手该系统的其他部分。我开发了一个调试器（debugger，调试程序、排错程序），我把它称为GDB（译注：GNU调试程序的简称），那是个用于C代码的符号调试程序，最近已开始分发。现在，这个调试程序预计会是DBX精神（spirit，潮流、风格）中的主要部分，DBX是一个始于Berkeley（伯克利，一般指加利福利亚大学伯克利分院）Unix的调试程序。指令都由一个跟有变元（arguments，自变量、参数）的、说明你要做什么的单词构成。在这个调试程序中，指令可以全部被化简，而且常用指令都有单一的字符缩写，但任何唯一的缩写总是被允许的。有可扩展的“HELP”（帮助）工具，你可以键入HELP，后跟任何指令甚至子命令，就获得一长段如何使用该指令的描述。当然你也可以敲入C语言中的任何表达式，而它将显示其值。你也可以用C符号调试器做一些不常见的事，例如：你可以查阅在任何存储地址的任何C数据格式，还可测值或赋值。这样的例子有，如果你想用一个词（命令）在某特定地址存储一浮点值，你只需说（用该命令表示出这样的意思）：“在这个地址给我类型为FLOAT（浮点）或DOUBLE（双精度）的对象”，这就指定了该值类型。你可以做的另一件事是检测所有先前被检测过的值。每个被检测过的值都被记入到“值历史”。你可以通过其号数位置，在历史中查阅任何元素（element），或者，你可以简单地键入美元符号查看最近一次的元素。而这使它很容易跟踪表结构。如果你有任何种类的、包含一个指向另一结构的C结构，你可以做像这样的事，键入命令：“PRINT *$.next”，这表示：“获取来自你给我显示过的上一个东西中的下一个域（field），然后显示它指向的结构。”你可以重复这个命令，而每次你都将看到当时表中的下一个结构。而在我见过的所有其他C调试器中，做到这点的唯一方法是每次都敲入一个较长的命令。而当这结合了只敲回车键来重复你发过的命令的特性时，它变得非常方便。对你想查看的表中每一元素，只敲回车键就行了。

调试器中也有直接可定位变量（settable variables），你键入美元符并跟一个名称，那就是指定了一个变量。你可以给这些变量赋与任何C数据类型的值，然后你可以检测它们。这些东西中适用的是：如果有个你要检测的特定值，而且你知道你要多次查看它，你可以给它一个名称，那比记住它在历史中的号数方便得多。在你设置条件断点时，你也可以发现适合的用途。条件断点是很多符号调试器中的一个属性，你说（发出的命令意思是）“在你达到程序中该点，且仅当一个特定表达式为真时，停止”。调试器中的变量允许你让（被调试）程序中的一个变量，与你已存入一个调试器变量中的、该变量的先前值作比较。另一件事是，它们可以被用于计量，由于各种赋值毕竟都是C中的表达式，因此你可以键入“$foo+=5”，以５为单位来增加“$foo”的值，或者你可以只敲“$foo++”。你甚至可以在一个条件断点中这样做，因此，你可以敲“$foo--==0”――这是一个让它中断到第十次而达到断点的轻巧办法。你们都明白了吗？耗减foo（译注：5个5个地耗减两次），而若它到达零，中断。接着你给$foo设定你想它跳过的数量（步长单位），你又让它继续运行。你也可以用其检测一个数组的元素。假设你有一个多指针数组，你可以键入：

 

PRINT X[$foo++]

 

但你首先要敲

 

SET $foo=0

 

好啦，当你这样做（原注：讲演者指向“Print”表达式）时，你得到X的第零个元素（译注：电脑中的第0个相当于日常说的第1个），接着，你再做就得到第一个元素，并假设这些都是指向结构的指针，那你可能得键入一个星号在那儿（原注：Print 表达式中的X的前面），而每次它都显示出被数组元素指向的下一个结构。当然你可以通过敲回车键来重复执行这一命令。如果重复一个单独的东西还不够用的话，那你可以创建一个用户自定义命令。你可以键入“Define Mumble（定义Mumble）”，接着你给出几行命令，然后键入“end” （结束）。此时，一个叫“Mumble” （咕哝自语）的命令就被定义了，它将执行那几行内容。把这些定义放进一个命令文件是非常有用的。你可以在每个目录中都有一个命令文件，当你开始起动调试器调试程序时，它将会按你的当前工作目录自动地装载相应的命令文件。因此，对每个程序你都可以定义一个用户自定义命令组，以一种好用的方式来存取（访问）该程序中的数据结构。你甚至能够为你的用户自定义命令提供一个用法说明文档，以便它们恰像那些内置命令一样，也能被“HELP”特性所操作。这个调试器中另一个有用的东西是从堆栈中清除已不用的栈框（frames，堆栈页框）。因为我相信它的重要性，不只是能够检测你正调试的程序中发生着什么，而且还能用任何可能想到的方法修改它。因此在你发现一个问题并知道错在何处后，你能够修复，仿佛代码被纠正，并且不必重编译你的程序，就能发现下一个毛病。这意味着不仅能够灵活地修改你程序中的数据存储区（data areas），而且能够修改控制流程。用这个调试器，你可以通过键入命令：

 

SET $PC=<some number>，

 

来非常便捷地改变控制流程。

这样你能指定程序计数器。你也可以指定堆栈指针，如果你想给栈指针一个特定的增量，那么你可以键入指令：

 

SET $SP+=<something>，

 

但是，此外，你还可以告诉它从该程序中特别的一行开始，你可以把程序计数器指定到一个源行。但如果你发现你错误地调用了一个函数，而你真想调用的根本不是那个函数，会怎样？瞧，那个函数被搞得如此之糟，使得你实际要做的是摆脱它返回到前面，再用手工去做那函数该完成的事。对此情况，你可以使用“RETURN” （返回）命令。你选择一个栈框并键入“RETURN”，它形成将被清除的该栈框――所有的东西都在其中，好像该函数此刻就正在返回，而你也可以指定它应返回的值。这并不继续执行；它假装那个返回发生了，然后再次停止该程序，这样你就能继续修改其他东西。而运用配成一体的所有这些东西，你就可以对一个程序中正进行的项目，拥有非常好的控制。

另一个稍微有趣的东西是：C语言中有串常量（string constants），如果你使用一个表达式中的串常量，你又用该调试器正在计算该表达式，会发生什么事？它不得不在你正调试的程序中创建一个串。好，它创建了。它在那被调试程序中建立一个对MALLOC（译注：命令或函数名；意为“存储分配”）的调用，让MALLOC运行，然后返回控制。这样，它暗中找到一个放置该串变量的地方。

 

最后，在这个调试器运行于真正的GNU系统时，我打算在该调试器中装入些应用工具，以检测正运行于其下的某过程（process）的所有内部状态。例如，要检测内存映射（译注：memory map，内存扩充利用技术，把一块或多块内存区的内容按使用情形复制到另一块可寻址内存区，以突破寻址局限，达到扩充复用等效果）的状态，哪些内存页（译注：page，被逻辑划分的内存使用块区）存在，哪些可读，哪些可写，并要检测次级程序的结束状态（internal status）。已经执行了一段命令后：跟UNIX上的调试器不一样，这个调试器保持让调试器和你正在调试的程序的结束状态彻底分开，以便它与运行于初级模式（raw mode）的程序一起工作，它与进行强制输入中断（interrupt driven input）的程序一起工作，而且这里也有一个命令，能使你找出你正在调试的程序中实际使用的与终态设置有关的东西。我认为，一个调试程序大体上应该允许你找出次级过程中发生着的每样东西。

在已经存在的GNU系统中还有另外两个主要部分。一个是新的C编译器，而另一个是TRIX内核。
